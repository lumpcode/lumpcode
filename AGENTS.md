# AGENTS.md
> Auto-maintained agent memory (Cursor continual learning). Dense, implementation-oriented — not user documentation; **commit to git** with repo changes so collaborators and hooks share the same context. See `packages/apps/cli/DOCS/` for public docs.

## Learned User Preferences

- Use "resumable" instead of "idempotent" when describing run behavior; preset agent command modules should persist agent chat/session id in `contextRunState` and across `keepHistory` so contexts can continue where they left off
- Prefer integration tests with real fixtures over mocking; avoid unnecessary dependency injection for testability (e.g., mock the underlying API instead); do not inject TTY detection unless there is a strong reason; test behavior, not implementation; unit tests that init temp git repos must set local `user.name`/`user.email` before `git commit` (CI runners lack global identity); tests that write files must use the temp `projectRoot`/`tmpDir`, not process `cwd`, so teardown removes them
- Use `cwd` option with `execAsync`/`child_process` instead of `cd ... &&`; command functions return only the command string, pass `cwd` at the call site
- Type functions with a direct `function` declaration or arrow functions; avoid `<Type>function` casts and the `const name = function name(...)` pattern
- One type per file in `packages/core/src/types/`; types defined independently from default implementations
- CLI conventions: unregistered `login`/`logout` command modules are **implementation-only**—do not document in user-facing README/DOCS (`npm login` in `DOCS/publishing.md` is npm registry auth only); arguments before options in usage; user-facing CLI docs (e.g. `COMMANDS.md`, task `*.desc.md`) use camelCase for long option names (e.g. `--lumpName`, `--contextName`, `--lines`) to match Commander/schema—avoid single-char option keys that need special `addCommand` handling (prefer `--lines` over `-n`)
- Keep user-facing CLI docs (`README`, `DOCS/`) aligned with real CLI behavior and terminology; avoid internal implementation detail (monorepo source paths, bundler/runtime stack, internal symbol names) unless an operator truly needs it. User-facing vocabulary: an **agent loop campaign** is the configured long-running work users call a **lump** (`lumpcode run`, `.lumpcode/lumps/`); say human review via **PR merge**, not vague "human review"; **LUMP** backronym is **Loop Using Multiple Prompts**; prefer periods/commas over em dashes in README/DOCS when not necessary; `concepts.md` intro is user-facing—put definitions in **Core terms**, not writer-only Terminology/meta sections. Use three workspace terms in prose: **project workspace** (`projectRoot`), **execution workspace** (`executionWorkspacePath` in CLI; git repo root after pre-flight), **branch workspace** (`workspacePath` on core `CommandFn` / `SetupWorkspaceFn` return—agent + per-context git cwd). **Tutorials** (e.g. `DOCS/get-started.md`) should be **self-contained**: readers can finish the guided path without **required** hops to other docs—links are for optional depth only. Document cross-lump **`dependsOnContexts`** in CLI DOCS only (`<otherLumpName>/<contextName>` slash syntax; context `name` must not contain `/`); `packages/core/README.md` stays core-only and links to CLI for cross-lump behavior. Root `README.md` includes an early-development disclaimer and `assets/lumpfish.png` after the lumpfish blockquote. Use the bare command name `"copilot"` (or `"cursor"`, `"aider"`, etc.) as the value of the lump-config `command` field — never include flags like `-p`, since the string is resolved as a registered command-module file name (`<configFolder>/commands/<name>.js`). Agent-specific flags belong inside the command module's `CommandFn` (`executable` + `args`). When CLI flags change, document only the current spelling—no migration guides or legacy argv tables unless the user explicitly asks for compatibility docs.
- `@lumpcode/cli` on npm is the primary public install path—user-facing README/DOCS document **`npm install -g @lumpcode/cli`** only (Node 22+); unscoped **`lumpcode`** meta package (`packages/apps/cli-meta`) may be published for `npm i -g lumpcode` but is not user-documented yet; standalone `install.sh`/`install-local.sh` and postinstall native-binary download are optional/build channels, not the launch install story—`postinstall` and standalone `install.sh`/`install-local.sh` reinstall bundled preset command modules to `~/.lumpcode/commands/presets/` (overwrite via `reset-presets`; sidecar `presets/` beside the binary is updated separately) and downloads the platform release artifact into gitignored `vendor/` (not published); skips both in CI, monorepo dev checkout (`src/root.ts` present), missing `dist/index.js`, or `--ignore-scripts`; `LUMPCODE_SKIP_BINARY=1` skips only native binary download (preset reinstall still runs); `lumpcode reset-presets` manually restores shipped defaults; tracked `bin/lumpcode.js` transparently runs the native binary when present else `node dist/index.js` with silent fallback on missing release assets; `clean` removes only SEA outputs under `bin/` (not the launcher)—root `.gitignore` ignores `packages/apps/cli/bin/lumpcode-*` and sidecars, not `lumpcode.js`; CLI `--version` reads `package.json` via static import in `main.ts` (no separate version constant); local debugging uses `build:dev` (core skips `.d.ts`; CLI ncc with source maps, no minify) then `NODE_OPTIONS='--enable-source-maps' node dist/index.js` from the target project cwd (absolute path to the monorepo entry works across repos)—not the SEA binary or npm launcher when it resolves to `vendor/`; SEA embeds minified `build:bundle`, so uncaught errors can print the entire one-line bundle as context; embed static assets (e.g. lump JSON schemas via static import into the ncc/SEA bundle) in the SEA binary when feasible; when npm and standalone installs may coexist, avoid PATH conflicts via `npx @lumpcode/cli`, npm alias install, or shell installer `--name lumpcode-beta`
- Limit change scope to specified packages; note follow-up work for other packages separately; keep CLI utils flat under `packages/apps/cli/src/utils/` (one util per directory with `main.ts` + `index.ts`, barrel-exported from `utils/index.ts`—no nested subdirs like `daemon/`); shared test-only helpers (e.g. `aliveDaemonSpawnFn` for daemon command integration tests) live under `packages/apps/cli/src/testing/` with a barrel `index.ts`, not under `utils/`
- Prefer single destructured object argument for functions with 3+ parameters for readability
- Resolve costly dynamic imports (`import()`) eagerly at configuration time, not lazily during execution
- Return `Success<T>` / `Failure<string>` result types instead of throwing for expected failures; use `success()` and `failure()` wrappers from `@lumpcode/core`
- Omit `repository` from publishable `package.json` files—it is not required for npm publish; postinstall native-binary download URLs (`native-binary.mjs` / `LUMPCODE_INSTALL_REPO`) are configured separately

## Learned Workspace Facts

- Monorepo (npm workspaces, **not** pnpm): `packages/core` (`@lumpcode/core`, Apache 2.0 engine on npm—`runLump` executes one **agent loop** per invocation over a context list), `packages/apps/cli` (`@lumpcode/cli`, Apache 2.0 CLI—primary public install via npm; manages agent loop **campaigns** in a repo: project setup, lump configs, daemon, status, cleanup), `packages/apps/cli/cli-types` (`@lumpcode/cli-types`: Apache 2.0 types and `define*` helpers for lump `config.js` and command modules; Rollup build is separate from the CLI SEA binary; pins `@lumpcode/core` to a semver range for npm, not `*`), `packages/libs/ui` (private WIP); TypeScript throughout with Commander.js for CLI parsing, Zod for validation, Vitest for tests; `packages/core/src/` laid out as `types/` (barrel-exported via `index.ts`), `usages/runLump/`, `helpers/`, `utils/`; agent-agnostic (Claude CLI, Codex, Aider, GitHub Copilot CLI `copilot`, or any CLI-based coding agent)—comparator is Codemod.com (AST-based, enterprise) vs Lumpcode's programmable TypeScript focus on individuals and small teams
- Core concepts: Project, Lump, Context (statuses derived from commit messages on remote refs—one normalized commit subject per context, supplied by `gitCommitMessageFn`; default core message `LUMP:${context.name}`, CLI uses `getGitCommitMessage({ contextName, lumpName })`; progression `toDo` → `branchPushed` → `finished` with the remote as source of truth: `toDo` when no matching commit exists on any remote ref, `finished` when a matching commit is an ancestor of `origin/<baseBranch>`, `branchPushed` when on some other remote-tracking branch; setting finished creates an empty commit on `baseBranch` and pushes it), Steps (recursive), Recipe; `getToDoContextList` validates context names via exported `validateContextListNames` (unique, `^[a-zA-Z0-9_-]+$`) before status checks; cross-lump `dependsOnContexts` uses composite `lumpName/contextName` and CLI `makeGitCommitMessageFnFromLumpName` resolves `/` in `context.name` to the referenced lump's marker (`LUMP:<referencedLump> - <contextName>`)—slash is only for dependency refs, not same-lump context names; per-lump `disabled` in `LumpJsConfig` may be a boolean, a zero-arg sync/async function, or a `FilePath` to a module exporting one—the daemon's `runTick` skips lumps that resolve as disabled; each lump loads **`config.json` or `config.js` only** (no `config.ts` runtime)
- CLI treats a directory as the Lumpcode project root when it contains both `.lumpcode` and `.git`; lump config (`LumpJsConfig` / JSON schema) omits `projectRoot`—the CLI sets engine `projectRoot` to the parent of `.lumpcode/` (`jsConfigToRunLumpInput` requires both `localConfigFolderPath` and `globalConfigFolderPath` and derives `projectRoot` from the former). **`.lumpcode/local.json`** (gitignored; scaffolded by `project-setup`, initial `mode` via `project-setup --mode`) is **required** for `run`/`start` with `mode` (`shared` | `dedicated`), `projectBaseBranch`, and optional `workspaceStrategy` (`checkout` | `worktree`, default `checkout`) and optional `disabled` (boolean—when `true`, daemon ticks skip all lumps on that machine without stopping the scheduler; manual `lumpcode run` is unaffected); `run`/`start` have **no** `--mode` or `--force`—change mode only by editing `local.json`. Pre-flight resolves **execution workspace** (`executionWorkspacePath`): project copy in `shared`, checkout in `dedicated`. Worktrees live under execution workspace at `.lumpcode/worktrees/<branch-as-nested-dirs>/`; worktree support stays CLI-only with engine `cwd` at source `projectRoot`—no core `gitRoot` / `workspaceRoot` API. Checkout-strategy setup/teardown use `atDirectory(executionWorkspacePath, …)` (`cd /d` on win32). Worktree strategy uses `git -C <executionWorkspacePath>` (not `cd && git` chains), `shellSingleQuote` on slash branch names, `shellBestEffort` for best-effort git/rm steps, platform-specific directory remove (`rm -rf` vs `if exist … rmdir /s /q`), and `mkdir` of the worktree parent before `git worktree add` on Windows. `makeLumpWorkspaceFns` maps **branch workspace** to core `workspacePath` on `SetupWorkspaceFn` return (`branchWorkspacePath` locals in CLI). `getCommandPath` resolves command modules only from explicit local/global config paths (no implicit `~/.lumpcode` fallback)
- This repo's development backlog is managed by the `todoStackPrds` lump: flat `TODO.yml` and `DONE.yml` under `.lumpcode/lumps/todoStackPrds/` (move entries to `DONE.yml` when finished); long-horizon ideas belong in root `IDEAS.yaml` (not parsed by the lump); each task has `name`, `task`, value-based `priority` (lower runs sooner), and `dependsOn` context names; `prds/<name>.prd.md` existence drives PRD-write vs implement contexts—no `prd:` YAML flag
- Core `executeStepsForContextList` resolves process `cwd` via `setupCommandCwd` / `workspaceExecCwd` from `projectRoot` and the `workspacePath` (branch workspace) returned by `setupWorkspaceFn`; recursive/dynamic `steps` walk-and-execute inline (each leaf runs `promptFn` → `commandFn` → command → `postCommandExecFn` before the next item resolves so `contextRunState` from prior steps is fresh); `CommandDescriptor` optional `env` merges over `process.env` for that spawn; `Step.continueOnError` (default false) lets a non-zero exit still run `postCommandExecFn` and continue the walk; `collectStepsForContext` is plan-preview only (`planLumpFromJsConfig`); `*-on-copy` presets keep `projectRoot` as the source repo and use an absolute copy path as branch `workspacePath` for setup, agent, git, and teardown; lump config `keepHistory: true` (CLI) wires `getKeepHistoryFilePathFn` to `.lumpcode/lumps/<lumpName>/history/<contextName>.json` under `projectRoot`—creating the history file runs `fs.mkdir(dirname(path), { recursive: true })` before the initial `[]` write so nested paths succeed; `project-setup` gitignores `.lumpcode/**/history/`
- Three mutually exclusive context-sourcing strategies: `contextListJson` (static JSON), `getContextListFn` (dynamic function), `contextMatchFn` (codebase file scanner—each per-path call receives `codeBasePath`, the full `codeBasePaths` array for that run, and `lumpVariables`; matches sharing a `contextName` merge into one context—variables accumulate, duplicate keys and `contextOptions` overridden by later matches)
- Default work-branch prefix is `lump/${lumpName}/` (internal `LUMP_BRANCH_PREFIX`); the public `branchPrefix` config option was removed—users needing custom naming pass `branchFn` explicitly (CLI default in `jsConfigToRunLumpInput`: one context → `lump/<lumpName>/<contextName>`; multiple contexts on one branch → sorted names joined with `\0`, SHA-256, first 12 hex chars). Default `gitPushCommandFn` pushes only the branch (no tag refs—context tracking is commit-message based). Default `gitCommitCommandFn` uses `git commit --allow-empty` (so no-op agent runs still create marker commits) and default `gitCommitCommandFn`/`gitPushCommandFn` wrap commit messages and branch names with `shellSingleQuote`. Of the four git*CommandFn fields inherited from `RunLumpInput` (`gitAddCommandFn`, `gitCommitCommandFn`, `gitCommitMessageFn`, `gitPushCommandFn`), only `gitCommitMessageFn` is the surfaced user knob (`LumpJsConfig` omits it via `Omit` so the CLI can default it through `getGitCommitMessage`); the rest are technically passable but not documented as user-facing hooks
- CLI daemon lifecycle: `start` detaches a background daemon by default (pass `--foreground` to block in the current terminal), with `stop`/`restart`/`daemon-log`/`daemon-status` companions; optional `--lumpName` on `start`/`stop`/`restart`/`daemon-log`/`daemon-status` scopes to one lump (per-lump PID/log/meta: `<projectName>.<lumpName>.daemon.*`; global: `<projectName>.daemon.*` under `<globalConfigFolderPath>/daemons/`). **PID and meta JSON are written only in `--foreground`** (detached parent spawns foreground `start` and must not write PID/meta itself). Scheduled ticks use Croner `{ protect: true }` with `await runTick()` so a long tick blocks the next cron fire (missed intermediate slots, not overlapping ticks); lumps run sequentially within a tick; daemon passes `lockMode: 'wait'` on `runLumpFromJsConfig` to queue behind branch-workspace locks held by other processes (e.g. manual `run`). `daemon-log` follows live by default (`tail -f`); `--noFollow` prints and exits; optional `--lines` limits initial output (still follows unless `--noFollow`). Starting the global daemon fails if **any** project daemon is already running; starting a per-lump daemon fails if the global daemon is running, the same lump's daemon is running, or any **other** per-lump daemon is running when `workspaceStrategy` is not `worktree`. `local.json` is read once at daemon startup and frozen for all ticks (restart daemon to pick up changes). Per-lump `disabled` in lump config makes the daemon skip that lump on the next tick when it resolves truthy (there are no `lumpcode enable` / `lumpcode disable` subcommands—edit config to toggle). Meta JSON includes `cronSetup` (and optional `lumpName`) for `restart`/`daemon-status`; daemon files live outside the git working tree; `projectName` is read from `project.json`, must contain only letters, digits, `_`, and `-` (invalid names make the project unusable until fixed), and on `project-setup` it is inferred from `git remote get-url origin` (or sanitized repo directory basename) when `--projectName` is omitted—the CLI uses it as-is for those filenames and for `project-copies/<projectName>/`. In `shared` mode the daemon runs on that copy (source checkout untouched); in `dedicated` mode it runs in-place on the checkout (destructive pre-flight). **Pre-flight** (`runPreflight`) runs before every `run` and daemon tick: resolve execution workspace from `local.json` `mode`, then `git fetch` / switch / hard-reset / pull `projectBaseBranch` there (no `git status --porcelain` dirty-tree guard yet—`dedicated` mode can wipe uncommitted work on the real checkout). Treat `.lumpcode/` lump configs and command modules as trusted executable code in the repo. CLI `clean` uses `runProjectPreflight` so branch/worktree cleanup targets the same execution workspace as runs; deletes lump branches (local + remote) and removes matching worktrees under `.lumpcode/worktrees/`; `--lumpName` scopes to one lump, `--contextName` (requires `--lumpName`) scopes to a single context. `LumpJsConfig.maximumNumberOfConcurrentBranches` is enforced inside `runLumpFromJsConfig` via `countOpenLumpBranches` at `executionWorkspacePath` (`git ls-remote --heads origin` for `lump/<lumpName>/*`—remote only, local-only branches ignored); when the limit is reached the run returns a `skipped` variant instead of executing
- CI binary build (`.github/workflows/build-cli.yml`) runs a **`unit-test`** job on `ubuntu-latest` first (`npm i`, build `@lumpcode/core` and `@lumpcode/cli-types`—`cli-types` `dist/` is gitignored and preset/command-module tests import it—then `npm run test` for core and cli; Vitest only, not E2E); the OS **`build`** matrix has `needs: unit-test`; a final **`ci`** job (`needs: [unit-test, build]`) aggregates pass/fail for branch protection—GitHub **Main-Protection** ruleset should require **`ci`**, not individual matrix job names; `.github/CODEOWNERS` backs optional code-owner reviews. In SEA, `validateLumpJsonConfig` reads `schemas/lumpConfig.schema.json` beside `process.execPath` (and `presets/` for first-run preset install)—release artifacts must ship binary plus `schemas/` and `presets/` as siblings of the real binary path (or embed schemas in the bundle); a lone `PATH` symlink without those sidecars breaks JSON validation. - Command modules (in config `commands/` folders) export `command`, `setup`, `teardown`; local config folder takes precedence over global; canonical preset install is `installPresetCommands` (`ensurePresetCommandsInstalled/installPresetCommands.mjs`, discovered via `readdir` `*.js`); `ensurePresetCommandsInstalled` wraps it with SEA/npm bundle path resolution (no overwrite on first `run`/`start`/`lump-plan`); runtime resolution loads `~/.lumpcode/commands/presets/<name>.js` (not the binary sidecar directly)—first `run`/`start`/`lump-plan` copies missing files only; shipped preset `.js` files are plain ESM copied as-is (not bundled) and must not import `@lumpcode/core` or other npm packages unavailable under `~/.lumpcode/commands/presets/`—use Node builtins and relative `./utils/` helpers only; shipped `cursor`/`copilot` presets `.trim()` prompts, pass them unquoted to `-p`, and return `null` for whitespace-only prompts; both seed resumable sessions in `<commandName>Setup` (`cursorSetup` / `copilotSetup`) with `setupChatId`, `chatsIds`, and `chatIdIndex` / `newChat` step variables—cursor allocates via `cursor-agent create-chat` and passes `--resume`, copilot uses `randomUUID()` and `--session-id`. **Headless preset policy:** no user prompts (Cursor: `-p` + `--force` + `--trust` + `--workspace` + `--sandbox enabled`; Copilot: `-p` + `--allow-all-tools` + `--no-ask-user`—never `--yolo`/`--allow-all-paths`); writes scoped to branch `workspacePath`. - Commander global options (e.g. `--json`, `--verbose`) are defined on the root program; subcommands read them via `command.parent.opts()`, not from their own `argsAndOpts`. Command option schemas extend shared `baseCommandOptionsSchema` (`json` + `verbose`). `addCommand` registers optional booleans uniformly (same flag pattern for every `z.boolean().optional()`, including `json`/`verbose`—use Zod `.describe()` for `--help` text; avoid special-casing option keys); `lump-status` uses `--silent` for summary-only output. Omit the flag for the default; pass it once for the non-default—do not document or require `--<name> true|false`, and do not preserve legacy two-token boolean argv when changing CLI flags. **`cliLog`** emits the command **result envelope** only (`messages` + optional `data`); with `--json` it prints **one compact JSON envelope line** per invocation (`JSON.stringify` without pretty-print)—tests and parsers should expect a single line and fail loudly if none is found. **`addCommand`** calls injectable `exit(1)` on handler `Failure` and on Zod parse failure (JSON parse failures use the same envelope shape). **Operational logging:** shared core `Logger` (`createConsoleLogger`/`noopLogger`); optional `RunLumpInput.logger` (defaults from `verbose`); levels `error`/`warn`/`info`/`verbose`—`error` always prints (even with `--json`); `--json` suppresses other operational lines; `executeStepsForContextList` logs the active context at `logger.info` before each context's steps (prints without `--verbose`; suppressed with `--json`); CLI `--verbose` OR-merges lump-config `verbose`; `createCliLogger` prefixes `[lumpcode]`; daemon forwards `--verbose` to the foreground child; git commit/push soft failures in `executeStepsForContextList` log via `logger.error` + `formatExecFailureMessage`. Shell strings passed to `execAsync` (e.g. `git log --grep=…`) must escape user-controlled values (commit messages, context/lump names, branch names, `--format=` values) via the shared `shellSingleQuote` helper exported from `@lumpcode/core`—win32 uses cmd.exe double quotes (`""` for literal `"`); Unix uses POSIX single quotes—do not redefine locally; use `shellBestEffort` for best-effort fragments (`cmd || true` on Unix, `(cmd || cd .)` on Windows cmd.exe). Hook `contextRunState` is a single plain object created once per context inside `executeStepsForContextList` (`setupResult?.contextRunState ?? {}`); the engine never freezes, clones, or replaces it. Per-context lifecycle order is `branchFn` (once) → `setupWorkspaceFn` (once) → for each context: `setupFn` → prompt loop → `teardownFn` → `git add` + `git commit` per context → finally `git push` (once) and `teardownWorkspaceFn` (once). Lump config has **no** user-facing workspace setup hooks—the CLI generates workspace fns from `local.json` (`mode`, `workspaceStrategy`) and per-lump `baseBranch` (`makeLumpWorkspaceFns`); command-module `setup`/`teardown` compose after workspace setup in registration order
